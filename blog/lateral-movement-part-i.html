Lateral Movement: Part I

How a normal domain user with no admin privileges can exploit Group Policy Preferences (GPP) passwords to become local administrator across the organisation.

Introduction

In penetration testing engagements, one of the most critical phases after initial access is lateral movement. This is the process of moving through a network from one compromised host to another, escalating privileges along the way. In this first part of the lateral movement series, we focus on a well-known but still frequently exploited weakness: Group Policy Preferences (GPP) passwords.

What Are Group Policy Preferences?

Group Policy Preferences (GPP) is a feature in Active Directory that allows administrators to configure and deploy settings across domain-joined machines. One of its features allowed administrators to set local account passwords through Group Policy. The password was stored in an XML file on the SYSVOL share, encrypted with a known AES key.

The problem? Microsoft published the AES key in their MSDN documentation. This means any authenticated domain user can read the XML file from SYSVOL and decrypt the password.

The Attack

Step 1: Enumerate SYSVOL

As a normal domain user, you have read access to the SYSVOL share by default. Navigate to the Groups.xml file (or other policy preference XML files) that may contain stored credentials:

\\<DOMAIN>\SYSVOL\<DOMAIN>\Policies\{GUID}\Machine\Preferences\Groups\Groups.xml

Other common locations include:

  • Services\Services.xml (service account passwords)
  • ScheduledTasks\ScheduledTasks.xml (scheduled task credentials)
  • DataSources\DataSources.xml (database connection strings)
  • Drives\Drives.xml (mapped drive credentials)

Step 2: Extract the Encrypted Password

The XML file contains a cpassword attribute that holds the AES-256 encrypted password. This value looks like a Base64 encoded string.

Step 3: Decrypt the Password

Using the publicly known AES key, the password can be decrypted. Several tools automate this process:

  • gpp-decrypt (built into Kali Linux)
  • Get-GPPPassword (PowerSploit module)
  • Metasploit auxiliary module: post/windows/gather/credentials/gpp

Step 4: Lateral Movement

If the GPP password was used to set a local administrator password, this same password is now valid on every machine where the policy was applied. This means you can use it to authenticate to other domain machines using:

  • PsExec
  • WMI
  • WinRM
  • Pass-the-Hash techniques

From a single normal domain user account, you now have local administrator access across potentially the entire organisation.

Why Does This Still Work?

Microsoft released patch MS14-025 in 2014 that prevents new GPP password policies from being created. However, the patch does not remove existing XML files from SYSVOL. Many organisations that applied the patch never cleaned up the old policy files, leaving the credentials sitting in SYSVOL for years.

Remediation

  • Delete existing GPP XML files containing cpassword attributes from SYSVOL.
  • Apply MS14-025 if not already applied.
  • Use LAPS (Local Administrator Password Solution) to manage local admin passwords with unique, rotating credentials per machine.
  • Audit SYSVOL regularly for sensitive data exposure.
  • Change any passwords that were previously stored in GPP.

Conclusion

GPP password exploitation remains one of the quickest wins in internal penetration tests. It demonstrates how a single overlooked configuration can give an unprivileged user full control over an entire domain's workstations. In Part II of this series, we will explore additional lateral movement techniques for escalating further within a Windows domain environment.
